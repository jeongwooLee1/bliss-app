## 1. 어쩌다 이렇게 됐나 (근본 원인)

구멍은 사실상 **한 개**, 두 결함이 겹쳤다:
1. **공개키 노출(설계상 정상)** — Supabase publishable(anon) 키 `sb_publishable_3H-KTP0…`가 브라우저 JS 번들(`assets/index-BzbbIzQL.js`)에 평문으로 박혀 있다. 이건 Supabase 설계상 "원래 공개되는" 키다.
2. **RLS가 전부 "항상 열림"(치명적 결함)** — 거의 모든 테이블의 RLS 정책이 `USING(true)`(=인증 불문 누구나 전체 허용). 자물쇠(RLS)는 달려 있는데 전부 "항상 열림"으로 설정된 상태.

→ 결과: 앱 주소만 알면 누구나 브라우저로 `https://dpftlrsuqxqqeouwbfjd.supabase.co/rest/v1/customers?select=*` 같은 주소를 직접 쳐서 **페이지 단위로 전 테이블을 통째 다운로드** 가능했다. Supabase Advisor는 "RLS 켜짐=초록불"로 떠서 안전해 보였던 게 함정. 인증을 앱(JS) 레벨에만 의존하고 **DB 레벨(RLS) 게이트가 비어 있던 것**이 근본 원인.

전권(service_role) 키는 번들에 **없었다**(서버 전수 grep 확인) → 키로 인한 쓰기/관리자 탈취 경로는 아니고, **anon 읽기 노출**에 한정.

### 실제 유출(실제로 가져갔나) = 입증 불가 → assume-breach
**어떤 로그에서도 "외부 IP가 민감 테이블을 실제로 대량 read 했다"는 직접 증거를 찾지 못했다.** 단, 이건 "안전했다"가 아니라 **판별할 로그 자체가 없기 때문**이다:
- anon REST 호출은 브라우저 → `supabase.co`로 **직행**해 우리 서버(nginx)를 우회한다(14일치 nginx 로그에 `/rest/v1` 요청 0건 — 구조적으로 안 남음).
- DB 감사 로그(pgaudit) **미설치** → 어떤 SELECT가 있었는지 기록 없음.
- Supabase 로그(get_logs)는 **최근 24시간만** 보존 → 전부 차단 이후 시점이라 노출 기간 미포함.
- 접속이력(account_login_log)은 **6/1부터** 시작 → 노출 기간 3.5개월 중 대부분 미포함.

> **결론(정직하게):** "유출 없음"이 아니라 **"유출 여부를 가릴 증거가 존재하지 않음"**. 보안 사고 대응 원칙상 **흔적 없는 전면 노출 = 유출됐다고 가정**하고 대응하는 게 맞다.

## 3. ⚠️ 법적 고지 의무 (개인정보보호법 / PIPA)

약 **4만 고객 PII + 서명 동의서(신체부위·시술 차트 = 민감정보)** 가 3.5개월 노출됐고 유출 여부를 입증할 수 없는 상황이면, PIPA상 **개인정보 유출 통지·신고 의무**를 보수적으로 검토해야 한다:
- 정보주체(고객) 통지, 일정 규모/민감정보 포함 시 **개인정보보호위원회·KISA 신고**(인지 후 72시간 내 신고가 원칙).
- 규모(4만+)·민감정보(서명동의서) 고려 시 신고 대상일 가능성이 높음 → **초기에 개인정보 담당 전문가(변호사/법무) 자문**을 받는 것을 강력 권고. 이건 기술이 아니라 법적 판단 영역.

## 5. "인증된 IP만 DB 접근" — 왜 이게 핵심 답이 아닌가

직관적이지만 **이 구조엔 핵심 방어로 못 쓴다.** 데이터 출입문(Supabase REST/RPC)은 **손님·직원이 각자 휴대폰·집·카페·매장 와이파이 등 아무 데서나** 들어와야 하는 공개 정문이다. "이 IP만 허용" 명단을 만들면 손님·직원이 못 들어온다. 그래서 정문은 열려 있어야 하고, **"들어온 사람이 올바른 열쇠(세션 토큰)를 가졌는지"로 막는 게 맞다 — 이게 6/14에 한 RLS+토큰 게이트이고, 이미 올바른 방향.**

IP 제한이 **실제로 효과 있는 곳(보조적, 권장):**
- **Supabase Network Restrictions** — DB 직접 연결(5432, psql·백업 덤프)에만 적용. "백업/관리 PC IP만 직접 접속 허용"은 가능·권장. 단 이번 REST 사고는 못 막음.
- **오리진 서버(158.179.174.30)** — SSH(22)에 사무실/집 고정IP만 허용, 80/443은 **Cloudflare IP 대역만 허용**(CF 우회 차단). 이건 IP 기반이라 실효성 있음.

**한 줄:** "특정 IP만 허용"은 공개 API라 불가능·부적절. 진짜 방어 = 이미 한 '토큰 없으면 차단'을 빠진 곳(housebook·스토리지·Edge 함수)까지 확장 + IP 제한은 관리자 직접 DB접속·SSH·오리진에만 보조 적용.

## 6. 강화 로드맵 (우선순위)

### 즉시 (오늘~내일, CRITICAL)
1. **send-sms·send-alimtalk 인증을 공개키 → 별도 공유 시크릿(또는 세션검증)으로 교체** + 하드코딩 공개키 폴백 제거. (2~4h)

### 단기 (이번 주)
4. **Gemini 호출을 서버(bliss_naver.py)로 이관 + settings에서 키 제거** + 키 폐기/재발급(서버 보관). (1~2일)
5. **오리진 방화벽을 Cloudflare IP 대역만 허용** + SSH 고정IP 제한 + fail2ban + SSH root 로그인 off. (반나절)
6. **env.conf 600 + GitHub/Telegram/Gemini 토큰 로테이션.** (1h)
7. **bliss-uploads 버킷 비공개 전환 + 서명URL**(consents와 동일 패턴). (반나절)
8. **전 직원(22계정) 비번 강제 리셋**(자격증명 노출 대비) + accounts `password` 죽은 컬럼 제거 + 로그인 RPC 무차별대입 방어(N회 실패 잠금). (1일)
9. app_sessions 영구토큰(만료 NULL 4건) 정리 + 만료정책 강제. (반나절)

### 장기 (구조 개선)
10. **Supabase Network Restrictions**(5432 직결을 백업/관리 IP만). 
11. **PITR(시점복구)** 활성 여부 확인·활성화(변조/랜섬 대비 — housebook은 anon DELETE 권한이 실재했음).
12. SECURITY DEFINER 함수 `search_path` 고정(권한상승 차단) — 변경 시 `ai_golden` 회귀 게이트 통과 확인.

14. 시크릿 인벤토리 + 정기 로테이션 + 배포 파이프라인에 **번들 시크릿 grep 게이트**(service_role/AIza/ghp_/sk- 검출 시 빌드 실패) + 신규 테이블 RLS 기본잠금 체크리스트.
